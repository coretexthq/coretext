# Qualitative Comparison Report: File-Based vs Graph-Based Retrieval

**Experiment:** CoreText Evaluation — `trore` (Rental House Listing Platform)
**Date:** 2026-03-31
**Scope:** Epic 1, Stories 1-1 through 1-5 (Create-Story Phase)
**Subjects:** Experiment B (File-Based Control) vs Experiment C-2 (CoreText Graph-Based Test)

> **Companion document:** For token counts, request volumes, and efficiency metrics, see [quantitative-comparison.md](file:///Users/mac/Git/coretext/experiments/trore/results/comparison/quantitative-comparison.md).

---

## 1. Executive Summary

This report presents a qualitative analysis of **implementation story artifacts** generated by two architecturally different AI agents tasked with the same software specification. Where the quantitative analysis demonstrates that CoreText (Exp-C) achieved a **5.4% token efficiency gain** over the file-based baseline (Exp-B), this qualitative analysis reveals a more nuanced picture: **Exp-B consistently produced higher-fidelity artifacts** that more closely aligned with the project's architectural and design specifications.

### Headline Findings

| Dimension | Winner | Margin | Key Insight |
|:----------|:------:|:------:|:------------|
| **Architecture compliance** | Exp-B | Large (+10 to +17 per story) | Exp-C systematically missed global structural rules (feature-folders, state management, type pipelines) |
| **Epics fidelity** | Exp-B | Small (+1 to +6 per story) | Both preserved acceptance criteria well; Exp-C occasionally hallucinated extra scope |
| **PRD coverage** | Mixed | Story-dependent | Exp-C outperformed on Story 1-3 by cross-referencing FR3 from the PRD — a requirement Exp-B missed entirely |
| **UX specification** | Exp-B | Moderate (+1 to +6 per story) | Exp-C frequently dropped design system requirements (Tailwind CSS, color tokens) |
| **Cross-document reasoning** | Exp-C | Situational | Graph retrieval excels at surfacing requirements scattered across multiple documents |
| **Guardrail consistency** | Exp-B | Dominant | File-based reads act as a passive safety net — global rules are always present in context |

### The Core Tension

> **Exp-B trades efficiency for safety.** By loading entire spec files, it guarantees global constraints are always visible, but at higher token cost.
>
> **Exp-C trades safety for efficiency.** By retrieving only relevant subgraph nodes, it reduces token consumption, but risks missing cross-cutting architectural mandates that are semantically distant from the story being implemented.

## 3. Thematic Analysis

### 3.1 Architecture: The Structural Compliance Crisis

The architecture spec defines nine cross-cutting dimensions — from folder structure to error handling. Across all five stories, Exp-C exhibited a **consistent and repeating pattern** of failing to comply with three specific architectural mandates:

#### The "Architectural Triad" of Persistent Failures

| Dimension | Exp-B Average | Exp-C Average | Failure Pattern in Exp-C |
|:----------|:------------:|:------------:|:-------------------------|
| **A2: Feature-folder architecture** | 5.0 | 1.4 | Defaulted to `src/pages/` + `src/components/` in 4 of 5 stories |
| **A5: State management** | 5.0 | 2.2 | Missed TanStack Query in 4 of 5 stories; used raw `useEffect` instead |
| **A7: Type safety pipeline** | 4.8 | 3.0 | Skipped `packages/types` sync in 3 of 5 stories |

These three dimensions alone account for **42 of the 60-point architecture deficit**.

**Root Cause (from RCA investigations):** The architecture spec organizes these rules under global section headers like `§Structure Patterns` and `§State Management Patterns`. These sections do not contain story-specific keywords like "listing creation" or "admin dashboard." When Exp-C's agent formulates a `query_knowledge` call such as `"Architecture patterns for Admin Dashboard"`, the vector search returns nodes that are semantically close to "admin" — such as state management or PRD admin requirements — but misses the globally-scoped structural patterns that live under different headings entirely.

> [!WARNING]
> **The fundamental design tension:** Global constraints that apply to *every* story are the hardest for semantic search to retrieve, because they are *semantically equidistant* from all stories and therefore never in the top-k for any specific story query.

#### Story 1-5: The Convergence Effect

Story 1-5 produced the **narrowest gap** (7 points on Architecture, vs 11–17 for earlier stories). The RCA attributes this to Exp-C's graph being fully populated by this stage — including previously generated story artifacts (1-1 through 1-4) — which provided indirect architectural guidance through cross-references. This suggests **graph retrieval improves over time** as the knowledge base densifies.

---

### 3.2 PRD: The Cross-Reference Advantage

While Exp-B dominated on Architecture, Exp-C demonstrated a **unique strength** in PRD compliance — most notably on Story 1-3:

| Story | Dimension | Exp-B | Exp-C | What Happened |
|:------|:----------|:-----:|:-----:|:--------------|
| 1-3 | P1: Functional Requirements (FR3: Keyword Search) | **1** | **5** | Exp-B missed FR3 entirely. Exp-C found it via graph cross-referencing. |

**Root Cause:** FR3 ("Seekers can perform standard keyword searches") is defined in `prd.md` under `§Property Discovery & Search` and is *implicitly* mapped to Story 1.3 via the Epic 1 FR coverage map. However, `epics.md` under `§Story 1.3` only lists two acceptance criteria — neither of which mentions keyword search. 

- **Exp-B** read the full `epics.md` and followed the local story ACs rigidly, never cross-referencing back to the PRD's FR mapping.
- **Exp-C** queried `"Requirements for seeker discovery grid"`, which surfaced `prd.md#property-discovery-search` as a top-k result alongside the epics node. Because both nodes appeared with equal weight in the response, the agent treated FR3 as a requirement and implemented it.

> [!TIP]
> **The graph's "flattening" effect:** By collapsing hierarchical document boundaries into a flat list of semantically-retrieved nodes, graph retrieval can surface cross-document requirements that a file-based reader might miss when it follows a single document's internal hierarchy.

This advantage manifested as a **+2 PRD lead for Exp-C** on Story 1-3, and Exp-C matched Exp-B's PRD score on all other stories.

### 3.3 Epics: Fidelity vs Initiative

Both experiments maintained strong Epics alignment (98.7% vs 88.0%), but with qualitative divergence in failure mode:

| Behavior | Exp-B | Exp-C |
|:---------|:------|:------|
| **Story verbatim match** | Strict — copied stories word-for-word | Occasionally modified or paraphrased story descriptions |
| **AC preservation** | Exact Gherkin syntax, no additions | Added extra scenarios beyond what Epics defined (e.g., Story 1-3 added Keyword Search and Pagination ACs) |
| **Missing story handling** (1-5) | Conservatively inferred from FR20 | Aggressively expanded scope, adding deletion flow from nearest-neighbor nodes |

**Root Cause:** When Exp-C queried for story content that didn't exist (Story 1-5 is not explicitly in `epics.md`), the vector search returned **nearest-neighbor nodes** from other stories (e.g., `1-2-admin-manual-listing-creation.md`, `epics.md#story-3-2-listing-description-normalizer`). These tangential results polluted the context, causing the agent to hallucinate extended requirements to fill the gaps.

> **Interpretation:** Exp-C's initiative is a double-edged sword. When the graph surfaces genuinely relevant cross-cutting requirements (like FR3), the additional scope is valuable. When it surfaces noise from nearest-neighbor nodes, it leads to scope creep and requirement hallucination.

---

### 3.4 UX Design: The Chunking Problem

Both experiments scored well above 75% on UX alignment, but Exp-C exhibited a specific failure pattern related to the **design system foundation** (Tailwind CSS, Radix UI):

| Dimension | Exp-B Avg | Exp-C Avg | Notes |
|:----------|:--------:|:--------:|:------|
| U1: Design system (Tailwind + Radix) | 4.6 | 3.0 | Exp-C missed Tailwind CSS in 2 of 5 stories |
| U2: Color system (Trust Blue) | 3.5 | 3.5 | Both equally poor — treated as implicit via Tailwind config |
| U3: Component anatomy | 5.0 | 4.6 | Both strong |
| U4: Responsive strategy | 4.5 | 4.0 | Exp-C slightly less precise on breakpoints |
| U5: Feedback patterns | 4.5 | 4.8 | Exp-C slightly better on loading/empty states |

**Root Cause (Story 1-2 RCA):** When Exp-C retrieved `ux-design-specification.md`, the indexer returned it as a single monolithic file node rather than chunked by header. The entire document was delivered as one JSON string block in the `query_knowledge` response. The agent failed to extract specific design system details (Tailwind, Radix) from this dense, unstructured payload — effectively "having" the information but being unable to attend to the critical details.

> **Implication:** This is not a retrieval failure but a **presentation failure.** The chunking strategy of the knowledge graph directly impacts the agent's ability to extract actionable guidance from retrieved nodes.

## 5. The Efficiency-Quality Tradeoff

Combining the quantitative and qualitative data reveals a clear **inverse relationship** between token efficiency and spec alignment for the graph-based approach:

| Story | Exp-C Token Delta | Exp-C Quality Delta | Interpretation |
|:------|:-----------------:|:-------------------:|:---------------|
| 1-1 | **−25.8%** (more efficient) | **−14 pts** (lower quality) | Saved tokens but missed architecture |
| 1-2 | **−18.1%** (more efficient) | **−22 pts** (lower quality) | Fewest queries = most omissions |
| 1-3 | **+62.4%** (less efficient) | **−16 pts** (lower quality) | Spent more tokens searching but still missed architecture |
| 1-4 | **+10.8%** (less efficient) | **−18 pts** (lower quality) | Additional lint corrections consumed tokens without fixing architecture |
| 1-5 | **−30.6%** (more efficient) | **−8 pts** (lowest quality gap) | Graph maturity improved both efficiency *and* quality |

> [!IMPORTANT]
> **Story 1-5 demonstrates the convergence hypothesis:** As the knowledge graph densifies (more stories indexed, more cross-references established), the quality gap narrows while efficiency gains persist. This suggests the graph approach may become increasingly viable in later epics as the architectural context becomes embedded in the graph's topology.

## 6. Retrieval Method Analysis

### 6.1 Information Flow Comparison

```
Exp-B (File-Based):
┌─────────────────────────────────────────────────┐
│ read_file("architecture.md") → 604 lines        │
│ read_file("epics.md") → 395 lines               │
│ read_file("prd.md") → 156 lines                 │
│ read_file("ux-design-specification.md") → 190 lines │
│                                                   │
│ Total: ~1,345 lines of raw spec text in context  │
│ Coverage: 100% of all spec content               │
│ Relevance: ~20-30% directly applicable to story  │
└─────────────────────────────────────────────────┘

Exp-C (Graph-Based):
┌─────────────────────────────────────────────────┐
│ query_knowledge("story requirements...") → 5 nodes│
│ query_knowledge("architecture patterns...") → 5 nodes│
│                                                   │
│ Total: ~10 retrieved nodes (~100-200 lines)      │
│ Coverage: ~15-40% of relevant spec content       │
│ Relevance: ~60-80% directly applicable to story  │
└─────────────────────────────────────────────────┘
```

### 6.2 Retrieval Behavior Across Stories

From the [retrieved-knowledge-comparison.md](file:///Users/mac/Git/coretext/experiments/trore/results/comparison/retrieved-knowledge-comparison.md), we observe critical patterns in Exp-C's `query_knowledge` usage:

| Story | # Queries | Architecture Nodes Retrieved | Key Gap |
|:------|:---------:|:----------------------------:|:--------|
| 1-1 | 1 | `#requirements-to-structure-mapping`, `#implementation-handoff` | Missing `#structure-patterns`, `#state-management` |
| 1-2 | 2 | None from architecture.md | Complete architecture blind spot |
| 1-3 | 2 | None from architecture.md | Complete architecture blind spot |
| 1-4 | 2 | `#requirements-coverage-validation`, `#requirements-overview` | Missing `#structure-patterns`, `#state-management`, `#type-safety` |
| 1-5 | 3 | `#state-management-patterns` | Missing `#structure-patterns` — but state mgmt was found and applied |

**Pattern:** In Stories 1-2 and 1-3, Exp-C retrieved **zero architecture nodes**. The queries were entirely focused on the business feature, causing the vector search to return only Epics and PRD results. Story 1-5 shows partial recovery — `#state-management-patterns` was retrieved because the query included "Architecture patterns," and the agent successfully applied TanStack Query.

> **The missing query pattern:** No story ever successfully retrieved `architecture.md#structure-patterns` — the node governing the feature-folder mandate. This single retrieval gap is responsible for the most consistent failure across all five stories.

## 7. Strengths and Weaknesses Summary

### 7.1 Experiment B (File-Based) Strengths

1. **Architectural guardrails are passive and automatic** — by loading full files, every global rule is always present in context
2. **Zero architecture blind spots** — the agent always has access to the complete architectural specification
3. **Consistent structural compliance** — feature folders, state management, and type pipelines were correctly specified in every story
4. **Deterministic coverage** — no variance from run to run in what information is available

### 7.2 Experiment B (File-Based) Weaknesses

1. **Hierarchical tunnel vision** — tends to follow the local document hierarchy of one file without cross-referencing other documents (missed FR3 in Story 1-3)
2. **Higher token consumption** — loads ~1,345 lines of spec text regardless of what the story actually needs
3. **Diminishing returns at scale** — as the project grows, re-reading the same files every story becomes increasingly wasteful
4. **Implicit handling of cross-document dependencies** — relies on the agent independently noticing connections between files

### 7.3 Experiment C (Graph-Based) Strengths

1. **Cross-document requirement discovery** — the "flattening effect" surfaces requirements from multiple documents simultaneously, solving the FR3 gap that file-based retrieval missed
2. **Targeted precision** — when queries are well-formed, retrieves exactly the needed context with minimal noise
3. **Efficiency gains at maturity** — Story 1-5 demonstrated that as the graph densifies, both quality and efficiency improve
4. **Built-in quality enforcement** — the `lint` tool's referential integrity checks caught documentation errors the file-based agent never detected (Story 1-4)
5. **Lower token floor** — at peak performance (Story 1-5), achieved 66.1% better token efficiency

### 7.4 Experiment C (Graph-Based) Weaknesses

1. **Global constraint retrieval failure** — cross-cutting rules (folder structure, naming, state management) are semantically distant from any individual story and consistently fail to appear in top-k results
2. **Non-deterministic query generation** — the agent's natural-language queries vary in quality; a single poorly-formed query can cause cascading failures (Story 1-4: one query missed all architecture nodes)
3. **Nearest-neighbor pollution** — when searching for content that doesn't exist, the vector search returns tangentially related nodes that cause scope creep (Story 1-5's deletion hallucination)
4. **Chunking sensitivity** — improperly chunked documents (e.g., `ux-design-specification.md` as one monolithic node) reduce the agent's ability to extract specific details from retrieved content
5. **No architectural safety net** — unlike file-based retrieval, there is no mechanism to guarantee that global constraints are always present in the response

## 8. Recommendations

### 8.1 CoreText System Improvements

| Priority | Recommendation | Expected Impact |
|:--------:|:---------------|:----------------|
| **P0** | **Mandatory architecture pre-fetch:** The `create-story` workflow should always inject `architecture.md#structure-patterns` and `#state-management` nodes into context before the agent formulates queries | Eliminates the persistent feature-folder and state management failures (42 points recovered) |
| **P1** | **Improve document chunking:** Index `ux-design-specification.md` by header sections rather than as a monolithic file node | Improves Exp-C's UX alignment by enabling targeted extraction of design system rules |
| **P2** | **Add "global constraint" node category:** Create a special node type for cross-cutting rules that are automatically appended to all `query_knowledge` responses | Prevents global rules from being outranked by feature-specific nodes in top-k results |
| **P3** | **Enforce multi-query protocol:** Update the workflow to require at least 3 separate queries: (1) story requirements, (2) global architecture + tech stack, (3) UX/design constraints | Prevents the "kitchen sink" single query that drops important categories |

### 8.2 Spec Document Improvements

| Priority | Recommendation | Expected Impact |
|:--------:|:---------------|:----------------|
| **P1** | **Add FR mapping to story ACs in `epics.md`:** Story 1.3 should explicitly list FR3 in its acceptance criteria | Prevents both approaches from missing cross-document requirement mappings |
| **P1** | **Formalize Story 1.5 in `epics.md`:** The absence of an explicit Story 1.5 caused both agents to hallucinate scope | Eliminates nearest-neighbor pollution for Exp-C and scope ambiguity for Exp-B |
| **P2** | **Add explicit cross-reference anchors:** Each story in `epics.md` should link to the specific architecture sections it must comply with | Improves graph traversal by creating explicit edges to architectural constraints |

### 8.3 Workflow Improvements

| Priority | Recommendation | Expected Impact |
|:--------:|:---------------|:----------------|
| **P0** | **Add a "Spec Coverage Checklist"** to the create-story workflow: "Did you retrieve/read: Architecture? Epics? PRD? UX?" | Enforces minimum spec coverage before artifact generation |
| **P1** | **Forbid single-query concatenation in graph mode:** The workflow should explicitly instruct agents to never combine multiple intents into one `query_knowledge` call | Prevents the top-k dilution effect observed in Stories 1-1 and 1-3 |
| **P2** | **Dynamic `top_k` adjustment:** Allow agents to increase `top_k` from 5 to 10 when performing broad architectural queries | Increases the probability of retrieving semantically distant but critical global nodes |

---

## 9. Conclusion

The qualitative evidence demonstrates that **file-based retrieval (Exp-B) produces significantly higher-quality implementation artifacts** (92.7% vs 75.9% spec alignment), while the quantitative data shows that **graph-based retrieval (Exp-C) is more token-efficient** (5.4% cumulative gain).

This is not a binary verdict. The two approaches exhibit **complementary strengths:**

- **Exp-B excels** at enforcing global, cross-cutting constraints that must apply uniformly across all stories — the architectural "guardrails" that define a project's technical identity.
- **Exp-C excels** at discovering cross-document functional requirements that span multiple specification files — the kind of implicit mappings that a hierarchical file reader can miss.

**The optimal path forward is a hybrid approach:** use graph-based retrieval for focused functional requirements, but guarantee global architectural context through either mandatory pre-fetch or "sticky" constraint nodes that are always present in query responses. Story 1-5's convergence (−30.6% tokens with only −8 quality points) suggests this hybrid model becomes increasingly viable as the knowledge graph matures.
